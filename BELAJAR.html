🧱 Q&A Konsep OOP OOP

1. Apa itu Encapsulation dan tunjukkan di kode mana?

Encapsulation = menyembunyikan data di balik antarmuka publik (getter/setter). Akses langsung ke field dilarang dari luar kelas.

Contoh di kode: kelas SummaryRecord:

private final String originalText;
private final String summary;
// ... field semua private final

public String getOriginalText() { return originalText; }
public String getSummary()      { return summary; }

Field bersifat private final sehingga tidak bisa diubah dari luar (immutable) — bentuk encapsulation yang paling ketat.

2. Apa itu Inheritance? Kelas mana yang inherit?

Inheritance = kelas anak mewarisi field & method kelas induk dengan kata kunci extends.

Contoh:

  • RuleBasedSummarizer extends AbstractSummarizer
  • ApiBasedSummarizer extends AbstractSummarizer

Kedua kelas anak mewarisi field method, setter setMethod(), dan helper splitSentences() dari induknya.

3. Apa itu Polymorphism? Contoh konkret di kode?

Polymorphism = satu antarmuka, banyak bentuk implementasi. Pemanggil tidak peduli implementasi yang mana.

Contoh di MainFrame.buildSummarizer():

Summarizer summarizer = buildSummarizer();   // bisa Rule atau API
String result = summarizer.summarize(text);  // dipanggil sama caranya

Method summarize() dipanggil dengan cara yang sama walaupun objek aslinya bisa RuleBasedSummarizer ataupun ApiBasedSummarizer. JVM yang memilih implementasi yang tepat saat runtime — disebut dynamic dispatch.

4. Apa itu Interface? Tunjukkan di kode!

Interface = kontrak murni — hanya mendefinisikan signature method, tanpa implementasi. Kelas yang implements wajib menyediakan implementasi semua method-nya.

Contoh: Summarizer.java

public interface Summarizer {
    String summarize(String text) throws SummarizationException;
    String getName();
}

Dan Exporter.java dengan 3 method (export, getExtension, getDescription).

5. Apa beda Interface vs Abstract Class? Kenapa pakai dua-duanya?
InterfaceAbstract Class
Hanya kontrak (signature) Boleh punya field & implementasi sebagian
Satu kelas bisa implements banyak interface Hanya bisa extends satu abstract class
Untuk mendefinisikan "kemampuan" / "peran" Untuk berbagi kode antar kelas yang serumpun

Di proyek kita: Summarizer = interface (kontrak), AbstractSummarizer = abstract class (kode bersama). Pola ini lazim — interface dipakai oleh caller, abstract class dipakai implementor agar tidak menulis ulang kode yang sama.

6. Apa itu Enum dan mengapa SummaryMethod pakai enum?

Enum = tipe data dengan jumlah nilai tetap dan terbatas, type-safe.

Daripada pakai int (1 = pendek, 2 = sedang) yang rawan salah, enum memaksa caller hanya memilih dari nilai yang valid:

public enum SummaryMethod {
    SHORT("Ringkasan Pendek (2 kalimat)", 2),
    MEDIUM("Ringkasan Sedang (4 kalimat)", 4),
    DETAILED("Ringkasan Detail (6 kalimat)", 6);
}

Tiap enum membawa data (displayName & targetSentences) — fitur OOP yang lebih kuat daripada enum di bahasa C.

7. Apa itu Abstraction? Tunjukkan!

Abstraction = menyembunyikan kompleksitas, hanya menampilkan apa yang penting.

Caller cukup tahu: "panggil summarize(text), dapat ringkasan." Tidak perlu tahu apakah itu pakai frekuensi kata, HTTP request, atau apa. Detail tersembunyi di belakang interface Summarizer.

8. Mengapa pakai package (paket)? Apa fungsinya?

Paket = pengelompokan kelas berdasarkan tanggung jawab. Manfaatnya:

  • Kode terorganisir & mudah dicari.
  • Menghindari konflik nama kelas.
  • Mengatur akses (paket default = hanya boleh diakses dari paket yang sama).

Di proyek kita ada 5 paket: core (algoritma), io (input/output), history (riwayat), ui (tampilan), dan root com.ringkasan (entry point).

🧮 Q&A Algoritma Summarization Algoritma

9. Bagaimana cara kerja Rule-based Summarizer?

Algoritma extractive summarization berbasis frekuensi kata, langkahnya:

  1. Pecah teks menjadi kalimat (regex tanda baca . ! ?).
  2. Tokenisasi kata, ubah ke lowercase.
  3. Hitung frekuensi tiap kata, abaikan stopword (yang, dan, di, …) & kata pendek < 3 huruf.
  4. Normalisasi frekuensi ke skala 0..1 (bagi dengan frekuensi maksimum).
  5. Skor tiap kalimat = jumlah skor kata penyusunnya.
  6. Ambil N kalimat ber-skor tertinggi, urutkan kembali sesuai urutan asli.

Intuisi: kalimat yang banyak mengandung kata-kata sering muncul = kalimat paling representatif dari teks.

10. Apa itu stopword dan kenapa diabaikan?

Stopword = kata-kata umum yang sangat sering muncul tapi tidak bermakna untuk topik teks (mis. "yang", "dan", "di", "the", "is", "a").

Kalau dimasukkan, kata-kata ini akan mendominasi skor padahal tidak menggambarkan topik. Dengan diabaikan, skor lebih akurat menggambarkan kata-kata kunci sebenarnya.

11. Kenapa frekuensi dinormalisasi?

Agar skor antar kata berada di skala yang sama (0..1). Tanpa normalisasi, kalimat panjang otomatis dapat skor lebih tinggi karena punya lebih banyak kata. Normalisasi membuat relevansi kata lebih berperan daripada jumlah kata.

12. Kekurangan algoritma rule-based ini?
  • Hasilnya extractive — hanya mengambil kalimat asli, tidak menulis ulang.
  • Tidak paham makna semantik, hanya frekuensi.
  • Kalau teks pendek atau topik sangat beragam, hasil kurang akurat.
  • Tidak mengenali sinonim (mis. "mobil" dan "kendaraan" dihitung beda).

Inilah alasan kenapa API-based (HuggingFace, model BART) biasanya lebih baik — model AI memahami konteks dan menulis ulang.

13. Apa itu extractive vs abstractive summarization?
ExtractiveAbstractive
Memilih kalimat asli dari teks Menulis ringkasan baru (parafrase)
Mudah, cepat, no-AI Butuh model AI (BART, GPT, T5)
Hasilnya kaku, tapi pasti faktual Hasilnya natural, tapi bisa "halusinasi"

Rule-based di proyek = extractive. API HuggingFace BART = abstractive.

🪟 Q&A GUI Swing GUI

14. Kenapa pakai Swing, bukan JavaFX?
  • Swing built-in di JDK — tidak butuh dependency tambahan.
  • Portable: tinggal kompilasi & jalan di komputer manapun yang punya JRE.
  • Cocok untuk tugas kuliah yang fokus ke OOP, bukan estetika.

JavaFX harus di-download terpisah dan setup library — lebih ribet untuk demo.

15. Apa itu EDT (Event Dispatch Thread)?

EDT = thread khusus di Swing yang menangani semua event UI (klik tombol, repaint, dsb). Semua perubahan komponen Swing wajib dilakukan di EDT, kalau tidak bisa terjadi race condition / UI freeze.

Itu sebabnya di MainApp dipakai SwingUtilities.invokeLater(...):

SwingUtilities.invokeLater(() -> new MainFrame().setVisible(true));
16. Kenapa proses summarize pakai SwingWorker?

Karena proses API bisa makan waktu beberapa detik. Kalau dipanggil langsung di EDT, UI akan freeze (tombol tidak bisa diklik, jendela tampak "hang").

SwingWorker menjalankan pekerjaan berat di background thread, lalu hasilnya dikirim balik ke EDT lewat done():

SwingWorker<String, Void> worker = new SwingWorker<>() {
    protected String doInBackground() throws Exception {
        return summarizer.summarize(text);   // ← di luar EDT
    }
    protected void done() {
        outputArea.setText(get());           // ← kembali ke EDT, aman
    }
};
17. Apa itu Layout Manager? Mana yang dipakai?

Layout manager mengatur posisi & ukuran komponen otomatis sesuai ukuran window. Yang dipakai di proyek:

  • BorderLayout — bagi frame jadi 5 area (North, South, East, West, Center).
  • GridBagLayout — untuk panel pengaturan atas (paling fleksibel).
  • FlowLayout — untuk barisan tombol.
  • JSplitPane — pemisah resizable antara input & output.
18. Apa fungsi JFileChooser?

Komponen Swing untuk dialog pilih/simpan file standar OS. Dipakai di onLoadFile() (buka .txt) dan onSave() (simpan ke .txt/.pdf).

FileNameExtensionFilter dipakai untuk membatasi tipe file yang ditampilkan.

🌐 Q&A Integrasi API HuggingFace API

19. Bagaimana cara aplikasi memanggil API HuggingFace?

Pakai HttpURLConnection bawaan JDK (tidak perlu library tambahan).

  1. Buat URL endpoint: https://api-inference.huggingface.co/models/facebook/bart-large-cnn
  2. Method POST, header Authorization: Bearer hf_xxx
  3. Body JSON: {"inputs":"teks","parameters":{"min_length":50,"max_length":150}}
  4. Baca response stream → parse JSON ambil field summary_text
20. Apa itu Bearer Token / Authorization header?

Bearer token = string rahasia (kayak "tiket") yang membuktikan identitas user ke API. HuggingFace mengirimkan token saat user mendaftar. Tanpa token, request ditolak dengan HTTP 401 Unauthorized.

Format header: Authorization: Bearer hf_xxxxxxxxxxxxx

21. Mengapa pakai parser JSON manual dengan regex, bukan library?

Agar proyek tidak butuh dependency eksternal — cukup JDK murni. Respons HuggingFace strukturnya sederhana dan terprediksi:

[{"summary_text": "hasil ringkasannya di sini"}]

Cukup pakai regex untuk mengekstrak field summary_text. Tidak ideal untuk JSON kompleks, tapi cukup untuk use case spesifik ini.

Kalau project butuh handling JSON kompleks, baru pakai library seperti Jackson atau Gson.

22. Apa yang terjadi kalau API gagal?

Aplikasi melempar SummarizationException dengan pesan yang jelas:

  • HTTP 401 → "token tidak valid"
  • HTTP 503 → "model sedang dimuat, coba lagi ~20 detik"
  • Network error → "Gagal memanggil API: …"

Pesan ditampilkan ke user via JOptionPane.showMessageDialog() sehingga user tahu apa yang harus dilakukan.

📄 Q&A Ekspor PDF Algoritma

23. Bagaimana bikin PDF tanpa library?

PDF sebenarnya hanya file teks dengan struktur khusus (spesifikasi PDF 1.4). Komponen wajib minimal:

  1. Header — %PDF-1.4
  2. Catalog object — root dokumen
  3. Pages object — daftar halaman
  4. Page object per halaman
  5. Content stream — instruksi gambar teks (BT … ET)
  6. Font object — Helvetica (built-in PDF, gratis)
  7. xref table — index byte offset tiap object
  8. Trailer + %%EOF

Semua disusun manual sebagai string, lalu ditulis ke file. Lihat PdfExporter.java.

24. Kenapa tidak pakai iText / PDFBox?

Karena tidak butuh dependency tambahan = lebih portable, JAR lebih kecil, instalasi lebih cepat. Untuk PDF sederhana berisi teks, implementasi sendiri sudah cukup. Library besar dipakai kalau butuh tabel, gambar, atau font Unicode kompleks.

📁 Q&A Struktur Kode OOP

25. Kenapa proyek dipecah jadi 5 paket?

Prinsip Separation of Concerns — setiap paket punya tanggung jawab tunggal:

PaketTanggung Jawab
coreLogika summarization (rule-based & API)
ioInput (file/clipboard) & output (txt/pdf)
historyManajemen riwayat ringkasan
uiTampilan Swing (frame & dialog)

Struktur ini mendekati pola MVC — Model (core+history), View (ui), Controller (MainFrame).

26. Bagaimana cara menambah algoritma summarizer baru?

Cukup buat kelas baru yang extends AbstractSummarizer dan implement method summarize() + getName(). Tidak perlu mengubah kelas lain — ini contoh nyata Open/Closed Principle (terbuka untuk extension, tertutup untuk modification).

public class TfIdfSummarizer extends AbstractSummarizer {
    public String summarize(String text) { /* … */ }
    public String getName() { return "TF-IDF"; }
}

Lalu di MainFrame.buildSummarizer(), tambah cabang untuk metode baru.

27. Bagaimana cara menambah format ekspor baru (misal .docx)?

Implementasikan interface Exporter:

public class DocxExporter implements Exporter {
    public void export(String content, File file) { /* … */ }
    public String getExtension()    { return "docx"; }
    public String getDescription()  { return "Word Document (*.docx)"; }
}

Lalu tambahkan filter di MainFrame.onSave(). Kelas lama tidak perlu diubah.

28. Apa fungsi SummaryHistoryManager?

Menyimpan daftar SummaryRecord yang dibuat selama sesi. Sederhana, in-memory (ArrayList<SummaryRecord>).

API publik: add(), getAll(), getAllNewestFirst(), clear(), size(), isEmpty().

Pemisahan ini membuat MainFrame tidak perlu tahu detail penyimpanan — kalau besok mau pindah ke database, cukup ubah kelas ini.

🪤 Pertanyaan Sulit / Jebakan Tricky

29. "Kenapa di kelas SummaryRecord kamu pakai final untuk semua field?"

Karena saya ingin membuatnya immutable (tidak bisa diubah setelah dibuat). Manfaatnya: thread-safe otomatis, tidak ada bug akibat data berubah di tengah jalan, dan mudah dipahami. Riwayat ringkasan harus konsisten — entri lama tidak boleh berubah secara tidak sengaja.

30. "Kalau interface Summarizer punya method default, masih perlu AbstractSummarizer?"

Sejak Java 8, interface boleh punya default method. Tapi abstract class masih lebih baik untuk state (field method) karena interface tidak boleh punya field instance, hanya konstanta.

Aturan praktis: state pakai abstract class, behavior pakai interface.

31. "Method splitSentences() di abstract class — kenapa protected?"

Karena helper itu hanya berguna untuk kelas turunan, bukan untuk caller eksternal. protected = boleh diakses oleh subclass tapi tidak oleh kode dari paket lain yang tidak terkait. Lebih ketat dari public, lebih longgar dari private.

32. "Apa bedanya throws dan throw?"
  • throws = deklarasi pada signature method, memberi tahu caller bahwa method ini bisa melempar exception. Contoh: String summarize(String text) throws SummarizationException;
  • throw = perintah aktif untuk melempar exception. Contoh: throw new SummarizationException("Teks kosong.");
33. "Kenapa SummarizationException extends Exception, bukan RuntimeException?"

Karena ini checked exception — kondisi yang bisa diantisipasi dan wajib ditangani caller (input kosong, token invalid, jaringan down). Compiler memaksa caller pakai try-catch atau throws, sehingga kesalahan tidak terlupakan.

RuntimeException dipakai untuk bug programmer (mis. NPE, ArrayIndexOutOfBounds) yang seharusnya dicegah, bukan ditangani.

34. "Tolong demonstrasikan polymorphism HIDUP — ubah dropdown lalu jalankan."

Saat dropdown Metode diganti (Rule-based ↔ API-based), tombol RINGKAS memanggil kode yang sama persis. Polymorphism berlangsung di runtime — method summarize() dipanggil pada referensi Summarizer, JVM yang memilih implementasi yang tepat.

Demonya:

  1. Set Metode = Rule-based. Klik RINGKAS → hasil cepat, lokal.
  2. Set Metode = API-based + isi token. Klik RINGKAS → hasil dari HuggingFace.
  3. Tunjukkan di MainFrame kode summarizer.summarize(text) — identik untuk kedua kasus.
35. "Apa kekurangan aplikasimu? Apa yang bisa diperbaiki?"

Jawaban jujur menunjukkan kedewasaan teknis. Contoh:

  • Riwayat hanya in-memory, hilang saat aplikasi ditutup. Bisa dipersistenkan ke file/SQLite.
  • Rule-based belum dukung sinonim (mobil ≠ kendaraan).
  • PDF belum dukung font Unicode kompleks (mis. emoji).
  • Token API tersimpan plain text di field — idealnya disimpan di file config terenkripsi.
  • Belum ada unit test.

Dosen biasanya senang dengan jawaban seperti ini — menunjukkan kamu paham keterbatasan & jalan untuk improve.
