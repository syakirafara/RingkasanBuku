❓ Jawaban 4 Pertanyaan Dosen

1️⃣ Mengapa desain ini dipilih?

Desain dengan 16 tipe yang dipecah ke 4 paket dipilih dengan tiga pertimbangan utama:

  1. Separation of Concerns (Pemisahan Tanggung Jawab):
    • core = logika summarization murni
    • io = input/output (file, clipboard, ekspor)
    • history = manajemen data riwayat
    • ui = tampilan Swing
    Perubahan pada UI tidak menyentuh algoritma. Begitu pula sebaliknya.
  2. Extensible (Open/Closed Principle): menambah algoritma summarizer baru cukup membuat satu kelas baru yang extends AbstractSummarizer. Menambah format ekspor baru cukup implements Exporter. Kelas lain tidak perlu diubah.
  3. Maintainable & Reusable: komponen seperti SummaryFormatter, TextInputHandler, dan SummaryHistoryManager bisa digunakan ulang di kelas manapun.

Penggunaan 2 interface (Summarizer & Exporter) dan 1 abstract class (AbstractSummarizer) dipilih sebagai best practice — interface untuk "kemampuan/role", abstract class untuk berbagi implementasi.

2️⃣ Mengapa AbstractSummarizer menjadi superclass?

Saat menganalisis RuleBasedSummarizer dan ApiBasedSummarizer, ditemukan bahwa keduanya berbagi:

  • Field method bertipe SummaryMethod (panjang ringkasan).
  • Setter setMethod() dengan validasi null.
  • Helper splitSentences() untuk memecah teks ke kalimat.

Untuk menghindari duplikasi kode (prinsip DRY — Don't Repeat Yourself), state & helper bersama tersebut diangkat ke parent class AbstractSummarizer. Subclass cukup fokus pada algoritma uniknya:

public class RuleBasedSummarizer extends AbstractSummarizer {
    public String summarize(String text) { /* algoritma frekuensi kata */ }
}
📌 Aturan praktis: "Bila 2+ kelas berbagi field/method yang sama persis, angkat ke parent class."

3️⃣ Mengapa menggunakan inheritance?

Inheritance dipilih untuk menyatakan hubungan "is-a" & menghindari duplikasi kode. Di proyek ini inheritance terjadi di:

  • RuleBasedSummarizer extends AbstractSummarizer — RuleBased adalah sebuah Summarizer.
  • ApiBasedSummarizer extends AbstractSummarizer — ApiBased adalah sebuah Summarizer.
  • SummarizationException extends Exception — exception kustom kita masih punya tipe parent Exception.
  • MainFrame extends JFrame, HistoryDialog extends JDialog — kelas UI kita meminjam semua perilaku jendela Swing.

Tiga manfaat konkret:

  1. Code reuse: helper splitSentences() ditulis sekali, dipakai 2 kali.
  2. Konsistensi API: semua subclass otomatis memiliki setMethod, getMethod — tidak mungkin terlewat.
  3. Polymorphism: objek subclass dapat diperlakukan sebagai parent — inilah yang memungkinkan MainFrame menggunakan tipe Summarizer tanpa peduli implementasi konkretnya.

4️⃣ Apakah ada interface? Berapa?

Ada dua interface di proyek ini:

A. Summarizer (paket core) — kontrak summarization:

public interface Summarizer {
    String summarize(String text) throws SummarizationException;
    String getName();
}

Diimplementasi (lewat AbstractSummarizer) oleh RuleBasedSummarizer dan ApiBasedSummarizer.

B. Exporter (paket io) — kontrak ekspor file:

public interface Exporter {
    void export(String content, File file) throws IOException;
    String getExtension();
    String getDescription();
}

Diimplementasi langsung oleh TxtExporter dan PdfExporter.

Mengapa interface, bukan langsung abstract class?

  • Interface = "kemampuan/peran" (apa yang bisa dilakukan).
  • Abstract class = "implementasi bersama" (kode yang dipakai berbarengan).
  • Memisahkan keduanya membuat sistem fleksibel — kelas lain bisa implements Summarizer tanpa wajib mewarisi AbstractSummarizer jika tidak butuh helper bersama.

🗣️ Pernyataan Singkat (Saat Presentasi)

"Aplikasi saya terdiri dari 16 tipe Java — 14 class (termasuk 1 abstract), 2 interface, dan 1 enum — yang dipisahkan ke dalam 4 paket sesuai tanggung jawabnya: core untuk logika summarization, io untuk input/output, history untuk data riwayat, dan ui untuk tampilan Swing.
